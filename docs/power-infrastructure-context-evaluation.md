# Power Infrastructure Context Evaluation

## Purpose

This document evaluates whether Power Infrastructure Context belongs in Prime
Observer and whether APS outage / PSPS data is a suitable first candidate
provider within that evidence domain.

This is a documentation and evaluation pass only.

It does not:

- implement APS integration
- implement solar integration
- change dashboard behavior
- change launchd behavior
- change Prime Observer application behavior

If a conclusion cannot be supported directly from repository evidence, it is
marked:

`Needs Matthew Review`

## Repository Constraints

Prime Observer is currently defined as:

- local-first
- deterministic
- explainable
- artifact-driven
- additive
- minimal-dependency
- graceful under failure

The repository also establishes these specific boundaries:

- Environmental Context is evidence only.
- Environmental Context must remain separate from Observation, Attribution,
  Health, and Noticeability logic.
- Environmental Context must be generated by deterministic local Python code
  into local JSON artifacts for renderer consumption.
- Optional providers must fail safely with meaningful unavailable artifacts.
- Browser code must remain renderer-only and must not gain secrets or direct
  provider access.

These constraints are described in:

- `README.md`
- `DECISIONS.md`
- `ROADMAP.md`
- `docs/environmental-context.md`

## Evaluation Question

Two questions are evaluated here:

1. Does Power Infrastructure Context belong in Prime Observer?
2. Is APS outage / PSPS data a suitable first candidate provider in that
   domain?

## Domain Definition

Power Infrastructure Context means bounded external evidence about regional
electric utility conditions that may help reduce uncertainty around observed
network instability.

Power Infrastructure Context is distinct from Internet Conditions.

- Internet Conditions describes broad Internet infrastructure disruption or
  anomaly evidence outside the home network.
- Power Infrastructure Context would describe bounded utility or outage-related
  evidence outside the home network that may help explain why network equipment
  or access infrastructure could be unstable.

These are separate evidence domains. Power Infrastructure Context should not be
folded into Internet Conditions or used to widen the meaning of the existing
`internet_conditions.json` artifact.

In Prime Observer terms, this domain is acceptable only if it stays framed as:

- external evidence about conditions outside the local home network
- factual summary context near the relevant time window
- additive context that may help an operator judge whether broader conditions
  may align with upstream-looking or whole-home instability

It is not acceptable if it becomes:

- a causal explanation engine
- an outage predictor
- a recommendation engine
- an input to Attribution or Noticeability scoring
- a generic utility-monitoring product expansion

## Fit Against Environmental Context Acceptance Criteria

Power Infrastructure Context is a plausible fit for Prime Observer because it
can satisfy the repository's Environmental Context acceptance criteria if kept
bounded.

### Operator Value

Power-related context can help answer:

- What is happening?
  A regional outage or utility event may explain why local network equipment,
  ISP access equipment, or neighborhood infrastructure appear unstable.
- Why?
  It may reduce uncertainty around broad instability, but it must not claim
  that power conditions caused the network behavior.
- Is this unusual?
  It can add factual context that something broader than the local network was
  occurring, without replacing Pattern Awareness.

This is strongest when local evidence already suggests a broader or ambiguous
issue rather than a clear local LAN-only problem.

### Architecture Fit

Power Infrastructure Context fits the current architecture if it remains:

- a separate optional Python fetch/generation step
- a separate generated JSON artifact
- renderer-consumed evidence only
- explicitly unavailable when the provider cannot be used

This aligns with the existing NextDNS and Cloudflare Radar patterns.

### Safe Degradation

This domain can degrade safely if the artifact shape still makes sense when the
provider is unavailable, stale, missing, or unsupported.

The repository already prefers unavailable artifacts over dashboard or
investigation failure.

### Scope-Creep Risk

This domain has meaningful scope-creep risk.

It becomes a poor fit if it expands into:

- outage correlation claims
- neighborhood-specific causal assertions
- account-specific utility views
- device telemetry ingestion
- forecasting or recommendation behavior

The acceptable version is narrow: public or operator-configured summary
evidence only.

## Recommendation On Domain Fit

Recommendation:

Power Infrastructure Context does belong in Prime Observer as a future
Environmental Context evidence domain.

Reasoning:

- it matches the repository's existing Environmental Context model
- it can reduce operator uncertainty without changing core semantics
- it can remain additive, deterministic, artifact-driven, and fail-safe
- it answers a real observability question without widening Prime Observer into
  a general utility or incident platform

This recommendation is for the domain, not yet for a specific implementation.

## APS As First Candidate Provider

APS is the first candidate provider evaluated in this domain because the
repository already names APS only as a future candidate for Power
Infrastructure evidence.

That existing repository position is intentionally narrow and does not amount
to implementation approval.

### Why APS Is A Plausible Candidate

APS is a plausible first candidate if it provides:

- public or operator-accessible outage summary data
- a clear generation timestamp or event window
- bounded factual summaries that can be represented locally
- enough stability to support deterministic local Python generation
- a useful unavailable state when the source cannot be accessed

If those conditions hold, APS would fit the intended evidence shape better than
many ISP-status or social-feed style sources.

### Why APS Is Not Yet Automatically Approved

APS is not automatically approved as the first implementation because the
repository still requires provider evaluation before implementation.

Open concerns include:

- whether APS exposes data in a stable, automatable, non-brittle form
- whether the available data is public summary evidence rather than
  account-specific or interactive UI state
- whether the source supports bounded factual output without scraping drift
- whether the unavailable path is acceptable often enough to justify the
  maintenance cost

These provider-level concerns remain:

`Needs Matthew Review`

### APS Suitability Assessment

Current recommendation:

APS is a suitable first candidate provider to evaluate for Power Infrastructure
Context.

Current implementation recommendation:

Do not implement APS yet.

Why:

- the domain fits the repository
- APS is already repository-aligned as a named future candidate
- but provider-level implementation suitability still depends on the real data
  surface, stability, and artifact shape

## What A Good First Artifact Would Look Like

A future Power Infrastructure Context artifact would be repository-aligned if
it is:

- provider-specific
- local JSON
- summary-only
- time-bounded
- explicit about status
- explicit about unavailable conditions

Example characteristics:

- `provider`
- `generated_at`
- `status`
- `summary`
- bounded `items`
- `source_scope`
- `note`

This document does not define a final schema.

Any first schema design is:

`Needs Matthew Review`

## Explicit Non-Goals

This evaluation does not support:

- adding APS calls to browser code
- adding power context to Attribution logic
- adding power context to Noticeability scoring
- adding power context to Observation semantics
- adding solar-specific logic in this pass
- adding dashboard UI in this pass
- adding Historical Investigation context in this pass

## Recommendation Summary

### Domain

Power Infrastructure Context belongs in Prime Observer as a future
Environmental Context domain.

### First Candidate Provider

APS is a suitable first candidate provider to evaluate next in that domain.

### Implementation Decision

Do not implement APS in this pass.

Next step should remain a bounded provider-feasibility and artifact-shape
evaluation before any code changes.

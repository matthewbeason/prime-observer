# Health Dimensions Calibration

Phase 1 produced the proposal and calibration fixtures. Phase 2 implements the
first Python-owned evaluator in `bin/health_dimensions.py` and writes additive
artifact fields only. This document remains the calibration reference; dashboard
and investigation rendering changes are deferred to Phase 3.

The Phase 2 implementation does not remove legacy noticeability, does not call
OpenRouter, does not add a database, and does not rewrite historical snapshots.

## Incident Summary

The calibration incident was a resolver-provider path event where the configured
NextDNS primary resolver path was persistently high while the secondary resolver
path, general Internet probes, LAN gateway, environmental context, and household
experience were normal.

Observed local telemetry and diagnostics:

- NextDNS primary path was sustained at approximately `280-360 ms`.
- NextDNS secondary path was approximately `80-118 ms`.
- Quad9 and general Internet probes were mostly approximately `23-59 ms`, with
  one transient outlier.
- LAN gateway was normal.
- Packet loss was effectively zero.
- Cloudflare Radar showed no broad Cox anomaly.
- APS context was normal.
- No household symptoms were reported; work laptop and Mac mini appeared normal.
- External diagnostic evidence showed the primary route from Phoenix/Cox to
  `vultr-syd` at approximately `299-400 ms`.
- Secondary route went to `zepto-iad` at approximately `80-95 ms`.
- Nearby LAX NextDNS PoPs responded at approximately `18-29 ms`.
- The active NextDNS session used the secondary path.
- Direct `dig` timings corroborated probe latency.

Expected interpretation for calibration:

- technical condition: `severe`
- user impact: `low` or `not_observed`
- operational risk: `elevated`
- detection confidence: `high`
- attribution domain: `resolver_provider_path`
- attribution confidence: `high`
- dependency state: primary degraded, healthy active fallback
- redundancy state: `reduced`

## What Prime Observer Got Right

Current Prime Observer correctly preserved several important facts:

- Resolver probes are separated from general Internet probes by `target_class`.
- Gateway evidence remains separate from WAN evidence.
- Sustained bad evidence requires persistence, not a single excursion.
- Packet loss and latency remain visible as separate evidence.
- Optional provider context is supporting evidence and does not mutate telemetry.
- Investigation artifacts are generated by Python and rendered by the browser.
- Completed automatic snapshots are intended to be immutable and write-once.
- OpenRouter is not called by `bin/transform_latest.py`; transform only refreshes
  deterministic evidence and pending generation state.

## Current-Model Audit

### Technical Degradation Logic

Current WAN sample degradation is defined in `bin/health_model.py`:

- `WAN_BAD.p95 = 140 ms`
- `WAN_BAD.jitter = 50 ms`
- `WAN_BAD.loss = 1%`
- `WAN_BAD_PERSISTENCE = 2`

`is_wan_bad()` returns true when any WAN threshold is exceeded. In
`bin/transform_latest.py`, `mark_persistent_wan_bad()` marks `raw_bad` first and
then marks `is_bad` only after a consecutive bad streak reaches persistence. The
streak key is `(phase, target_class)`, so all resolver probes share one resolver
group streak and all Internet probes share one Internet group streak.

Buckets are 15 minutes. `classify_buckets()` reports raw bad counts, sustained
bad counts, p95/jitter/loss threshold driver counts, maximum raw bad run, and
`turbulence`. A bucket is turbulence when it has at least
`TURBULENCE_MIN_RAW_BAD = 4`, no sustained bad samples, and no raw bad run long
enough to satisfy persistence.

LAN elevation is separate. `lan_elevation()` marks LAN bad when at least three
gateway samples exceed `LAN_BAD_P95 = 120 ms` and the elevated rate is greater
than `20%`.

`bin/investigation_model.py` uses the same central WAN thresholds and recovery
constants. It selects automatic events from raw bad and sustained bad evidence,
computes lifecycle, severity, confidence, baseline/degradation/recovery windows,
scope facts, operator fallback fields, and immutable history payloads.

### Noticeability Inputs And Scoring

Noticeability is currently dashboard-only JavaScript in `viz/index.html`:

- `getNoticeabilitySummary()` reads the last 15 minutes of composite WAN samples.
- Inputs are median p95, p95-of-p95, count and fraction above `120 ms`, count
  above `180 ms`, and consecutive elevated p95 samples.
- It subtracts penalties from `100` and emits labels: `Probably masked`,
  `Low risk`, `Possibly noticeable`, or `Likely noticeable`.
- Forecast reads the last 60 minutes of composite WAN bad buckets, turbulence
  buckets, and WAN spike streak above `120 ms`; outputs are `Stable`, `Watch`,
  or `Risk`.
- `computeNoticeabilityForecast()` displays confidence from Pattern Awareness
  baseline confidence, not from noticeability evidence itself.

LAN evidence, active resolver path, application symptoms, user reports, DNS query
timings, route diagnostics, and optional environmental context are not current
noticeability inputs.

### Attribution Decision Tree

Current recent attribution is produced by `compute_recent_attribution()` in
`bin/transform_latest.py` and duplicated as fallback logic in `computeAttribution()`
in `viz/index.html`.

The last 15 minutes are considered:

- No recent WAN or LAN samples produces `No recent data`, low confidence.
- LAN bad plus WAN bad or WAN turbulence compares weighted WAN evidence points
  against LAN elevated samples.
- WAN dominance produces `Likely upstream (ISP / path)`, medium confidence.
- LAN dominance produces `Likely local (LAN / Wi-Fi)`, medium confidence.
- Unresolved concurrent LAN/WAN evidence produces `Mixed evidence`, medium
  confidence.
- LAN bad without WAN degradation produces `Likely local (LAN / Wi-Fi)`, high
  confidence.
- WAN sustained degradation without LAN bad produces `Likely upstream (ISP /
  path)`, high confidence.
- WAN turbulence without LAN bad produces `Likely upstream (ISP / path)`, medium
  confidence.
- No LAN or WAN degradation produces `No network issue detected`, high
  confidence.

Window attribution in `compute_window_attribution()` classifies sustained WAN
incident runs against LAN evidence during the same interval. Stable LAN makes the
incident likely upstream; bad LAN makes it likely local; missing or mixed LAN
evidence is inconclusive.

### Target Grouping

Current target grouping is in `bin/target_metadata.py`:

- `192.168.1.1` is `gateway_probe`.
- `1.1.1.1` and `9.9.9.9` are `internet_probe`.
- `45.90.28.134` and `45.90.30.134` are `resolver_probe`.
- Unknown targets are `unknown_probe`.

The model currently understands resolver probes as one target group, but does
not model resolver pairs, roles, active path, provider, PoP, fallback, or
per-member redundancy semantics.

### Confidence Semantics

Current confidence is a coarse label attached to specific outputs:

- Recent attribution uses `High` when evidence cleanly supports stable, local, or
  sustained WAN-upstream conclusions; `Medium` for turbulence or mixed dominated
  evidence; `Low` for no data or inconclusive.
- Window attribution confidence depends on consistency across incidents.
- Investigation event confidence is `high` when sustained bad count reaches
  persistence and `medium` otherwise.
- Bucket confidence is high for sustained or stable evidence, medium for raw bad
  or turbulence evidence.
- Noticeability confidence displayed on the dashboard comes from Pattern
  Awareness baseline sample confidence, not from user-impact evidence.

Current confidence does not distinguish detection confidence from attribution
confidence or user-impact confidence.

### JavaScript-Owned Semantics

The following semantics still exist in `viz/index.html`:

- WAN threshold constants, LAN threshold constants, persistence, turbulence, and
  bucket classification fallback.
- Dashboard row normalization and target metadata fallback.
- Composite WAN bucket fallback construction.
- Attribution fallback computation.
- Noticeability score, label, forecast, and confidence display.
- WAN health table statistics.

`viz/investigate.html` is closer to renderer-only. It detects legacy schema
versus schema 2, renders generated assessments, and has deterministic fallback
text for missing Operator Assistant output, but operational classifications,
scope, lifecycle, confidence, and freshness are Python-generated.

### Why This Incident Became High Noticeability

Noticeability used composite WAN p95 from both general Internet and resolver
probe samples. The severe primary resolver latency entered the same WAN
noticeability sample set even though the active resolver path was secondary and
user symptoms were not observed. Because the score penalizes elevated WAN p95,
severe p95, and consecutive elevated samples without dependency-group or active
path awareness, the incident could appear highly noticeable.

The root cause is dimensional conflation: technical resolver-path degradation
was treated as a proxy for user noticeability. The current score lacks active
fallback, user symptom, DNS query, and application-impact evidence.

### Why Attribution Stopped At Likely Upstream

Current attribution has only broad local versus upstream classes, with resolver
and Internet target groups used as evidence but not as attribution domains.
Stable LAN plus sustained resolver degradation maps to `Likely upstream (ISP /
path)`. The model cannot express `resolver_provider_path`, `upstream_transit_route`,
`resolver_endpoint_or_pop`, active fallback, or a provider anycast route problem.

## Proposed Dimensions

Phase 2 implementation note: the production evaluator uses the same dimensional
separation with production enum names aligned to `docs/health-model.md`:
`technical_condition` includes `elevated`, `user_impact` includes `unlikely` and
`likely`, and `operational_risk` uses `low`, `guarded`, `elevated`, `high`, and
`critical`.

The next model should emit separate dimensions. No dimension is a renamed
noticeability score. Each dimension should preserve its own evidence, missing
evidence, and confidence.

### `technical_condition`

Purpose: classify measured network or dependency degradation severity regardless
of whether a user noticed it.

Allowed states: `healthy`, `watch`, `degraded`, `severe`, `unknown`.

Required evidence: local telemetry samples for the relevant target or dependency
member, thresholds, persistence state, and sample freshness.

Optional evidence: direct DNS query measurements, traceroute summaries, resolver
route diagnostics, provider/PoP assignment, and environmental context.

Inputs: p95, jitter, loss, persistence, affected target group, per-member state,
DNS query timing when applicable, and protocol mismatch notes.

Exclusions: user reports, household symptoms, OpenRouter output, provider status
pages not tied to the target, and browser-only inference.

Confidence calculation: high when fresh telemetry has enough samples and
persistence; medium for fresh but sparse or protocol-mismatched evidence; low for
missing, stale, or legacy-only evidence.

Missing-evidence behavior: emit `unknown` when telemetry is absent; emit
`watch` rather than `degraded` for isolated outliers without persistence.

Explanation fields: `state`, `label`, `summary`, `drivers`, `thresholds`,
`affected_targets`, `sample_counts`, `missing_evidence`, `freshness`, and
`confidence`.

Interaction: may raise operational risk, may inform attribution, but must not
directly determine user impact.

### `user_impact`

Purpose: describe observed or likely human/application impact separately from
technical severity.

Allowed states: `not_observed`, `low`, `possible`, `confirmed`, `unknown`.

Required evidence: at least one of user report, application symptom, active path
state, or dependency-path evidence tied to a user-visible service.

Optional evidence: DNS query latency, endpoint fallback state, device-specific
operator observation, and application-specific probe evidence.

Inputs: active path health, fallback health, direct user reports, operator
observations, DNS query timings, application symptoms, and mismatch between ICMP
and DNS evidence.

Exclusions: raw resolver ICMP latency alone, broad environmental context alone,
and LLM interpretation.

Confidence calculation: high with confirmed symptoms and matching telemetry;
medium with explicit no-symptom report plus healthy active fallback; low when no
symptom or active-path evidence exists.

Missing-evidence behavior: emit `unknown` when no symptom or active-path evidence
exists; emit `not_observed` only when explicit no-symptom/operator observation is
fresh enough.

Explanation fields: `state`, `summary`, `observed_symptoms`, `active_path`,
`fallback_health`, `impact_drivers`, `impact_dampeners`, `missing_evidence`, and
`confidence`.

Interaction: can be `low` while technical condition is `severe`; can be
`confirmed` even when fallback appears healthy, but that must create unresolved
evidence rather than silently overriding telemetry.

### `operational_risk`

Purpose: express whether the operator should monitor, verify, or treat redundancy
as compromised.

Allowed states: `normal`, `elevated`, `high`, `unknown`.

Required evidence: technical condition plus dependency or scope evidence.

Optional evidence: active path state, fallback state, recovery trend,
environmental context, provider diagnostic references, and repeated history.

Inputs: severe/degraded technical condition, active path state, fallback health,
both-member degradation, redundancy state, persistence, unresolved mismatches,
and recovery state.

Exclusions: user impact alone without supporting telemetry, OpenRouter output,
and provider-neutral optional context that is stale or unrelated.

Confidence calculation: inherit detection confidence for technical facts and
reduce when dependency state or active path is unknown.

Missing-evidence behavior: if technical condition is severe but active path is
unknown, prefer `elevated` with missing active-path evidence; if both members are
degraded, prefer `high` even without symptom evidence.

Explanation fields: `state`, `summary`, `risk_drivers`, `risk_dampeners`,
`dependency_group_state`, `redundancy_state`, `missing_evidence`, and
`confidence`.

Interaction: uses technical condition and dependency state; does not redefine
attribution or user impact.

### `detection_confidence`

Purpose: rate confidence that Prime Observer detected the technical state from
available evidence.

Allowed states: `low`, `medium`, `high`.

Required evidence: telemetry sample count, freshness, thresholds, persistence,
and source provenance.

Optional evidence: corroborating direct measurements, diagnostics, baseline
context, and independent target groups.

Inputs: sample count, persistence, recentness, metric agreement, source count,
protocol agreement, artifact freshness, and schema completeness.

Exclusions: user report sentiment, LLM confidence, and attribution specificity.

Missing-evidence behavior: legacy artifacts without new fields are low; stale
diagnostics do not lower telemetry detection confidence but appear as unresolved
diagnostic evidence.

Explanation fields: `state`, `summary`, `sample_counts`, `freshness`,
`corroborating_sources`, `conflicts`, and `missing_evidence`.

Interaction: feeds attribution confidence and operational risk confidence but is
not itself attribution confidence.

### `attribution_confidence`

Purpose: rate confidence in the selected attribution domain.

Allowed states: `low`, `medium`, `high`.

Required evidence: target-group pattern, gateway evidence, and enough telemetry
to differentiate at least local versus non-local.

Optional evidence: traceroute summary, resolver route diagnostic, direct DNS
measurement, provider/PoP assignment, active dependency path, environmental
context, and operator observations.

Inputs: affected groups, unaffected comparison groups, LAN state, DNS-versus-ICMP
agreement, route or PoP evidence, environmental context, freshness, and
provenance.

Exclusions: user impact as causal proof, stale diagnostics as primary proof, and
provider-specific hard-coding.

Confidence calculation: high when telemetry and fresh diagnostics agree on a
specific domain; medium when telemetry narrows the domain but optional diagnostic
evidence is absent or stale; low when evidence is isolated, missing, or
conflicting.

Missing-evidence behavior: choose the most specific domain supported by telemetry
and mark unresolved evidence; use `unknown` when specificity is unsupported.

Explanation fields: `domain`, `summary`, `supporting_evidence`,
`evidence_against`, `unresolved_evidence`, `freshness`, and `confidence`.

Interaction: depends on technical condition and target-group evidence; does not
override technical facts.

## Attribution Taxonomy

Proposed deterministic attribution domains:

- `local_gateway`: gateway probe is persistently elevated and WAN degradation is
  contemporaneous or downstream of gateway elevation; stronger when wired and
  Wi-Fi gateway checks agree.
- `local_lan_or_wifi`: end-device or local access symptoms are present while
  gateway or wired comparison evidence narrows the issue to local LAN/Wi-Fi;
  requires more than WAN-only evidence.
- `broad_isp_path`: multiple non-local WAN target groups degrade together, LAN is
  stable, and environmental context does not show broad Internet/provider outage
  that better explains the pattern.
- `upstream_transit_route`: one or more remote routes beyond the access ISP show
  path-specific degradation, usually supported by traceroute summaries or
  route-hop divergence, while resolver/provider endpoint evidence is not the
  narrower explanation.
- `resolver_provider_path`: resolver target group is degraded while general
  Internet and LAN remain healthy, especially when resolver route diagnostics,
  active dependency path, or direct DNS timings corroborate a provider anycast
  path issue.
- `resolver_endpoint_or_pop`: diagnostics identify a specific resolver endpoint,
  PoP, or provider-assigned location as degraded while alternate PoPs or routes
  are healthy; stronger when nearby provider PoPs are healthy and the affected
  endpoint assignment is known.
- `broad_internet_condition`: multiple unrelated Internet evidence sources or
  Environmental Context indicate broad regional/provider-neutral degradation that
  aligns with local telemetry.
- `power_environmental`: power infrastructure evidence aligns with local network
  symptoms or gateway/WAN failure, with careful provenance and no provider status
  overreach.
- `mixed`: two or more domains have credible fresh evidence and neither
  dominates, or user symptoms conflict with healthy active fallback evidence.
- `unknown`: evidence is insufficient, stale, isolated, legacy-only, or
  contradictory.

Boundary rules:

- Broad ISP degradation means local gateway is stable and multiple non-provider-
  specific WAN targets degrade on the access path.
- Upstream route degradation means the issue appears beyond the access ISP or on
  a narrower route segment, usually requiring traceroute or route-summary
  evidence.
- Resolver-provider path degradation means resolver probes are affected while
  general Internet probes and LAN are healthy; it can be diagnosed without naming
  a specific PoP.
- Resolver endpoint or PoP degradation is narrower than resolver-provider path;
  it requires provider/PoP assignment or endpoint-specific diagnostic evidence.

## Dependency-Group Semantics

Resolver-pair modeling should be provider-neutral. The metadata should describe a
dependency group and members without hard-coding NextDNS.

Proposed metadata fields:

- `dependency_group_id`: stable local identifier, for example
  `resolver_pair_home`.
- `dependency_type`: enum such as `dns_resolver_pair`, later extensible.
- `member_id`: stable provider-neutral member identifier.
- `endpoint`: IP address, hostname, DoH URL, or other endpoint identifier.
- `role`: `primary`, `secondary`, `tertiary`, `peer`, or `unknown`.
- `provider`: display/provider string, optional but useful for explanation.
- `target_class`: normally `resolver_probe` for resolver members.
- `active_path_state`: `active`, `standby`, `inactive`, or `unknown`.
- `member_technical_condition`: `healthy`, `watch`, `degraded`, `severe`, or
  `unknown`.
- `fallback_health`: `usable`, `degraded`, `unusable`, `unknown`, or
  `not_applicable`.
- `redundancy_state`: `normal`, `reduced`, `no_usable_fallback`, or `unknown`.

Required computed dependency states:

- `both_healthy`
- `primary_degraded_secondary_healthy`
- `secondary_degraded_primary_healthy`
- `both_degraded`
- `active_path_healthy_peer_degraded`
- `active_path_degraded_fallback_healthy`
- `active_path_unknown`
- `redundancy_reduced`
- `no_usable_fallback`

Computed state guidance:

- Both members healthy means redundancy is normal.
- One member degraded while the peer is healthy means redundancy is reduced even
  if user impact is low.
- Active path healthy with peer degraded means user impact can be low while
  operational risk is elevated.
- Active path degraded with fallback healthy means user impact is possible and
  operational risk is high until failover is verified.
- Both members degraded means no usable fallback and high operational risk.
- Active path unknown must not assume low user impact; it should request operator
  input.

## Optional Diagnostic Evidence Schema

Optional diagnostic evidence should be additive, local-first, and provider-neutral.
It may refine impact or attribution confidence, but it must not silently override
telemetry.

Proposed artifact: `viz/diagnostic_evidence.json`.

High-level shape:

```json
{
  "schema_version": 1,
  "model_version": "prime_observer.diagnostic_evidence.v1",
  "generated_at": "2026-07-21T00:00:00+00:00",
  "status": "ok",
  "items": []
}
```

Shared item fields:

- `id`
- `type`
- `observed_at`
- `ingested_at`
- `freshness`: `fresh`, `stale`, `missing`, or `unknown`
- `freshness_seconds`
- `provenance`: `observed_fact`, `operator_assertion`, or
  `external_diagnostic_result`
- `source`: operator, script, provider diagnostic, imported file, or manual note
- `confidence`: `low`, `medium`, or `high`
- `target_association`: target host, target class, member ID, or dependency group
- `incident_association`: event ID, investigation ID, time interval, or null
- `summary`
- `details`
- `limitations`

Supported item types:

- `resolver_route_diagnostic`: resolver endpoint, assigned/observed PoP, route
  label, RTT range, provider-visible diagnostic reference.
- `direct_dns_query_measurement`: resolver endpoint, query name class, protocol,
  latency samples, response status, and measurement source.
- `traceroute_summary`: target, major hop/ASN/path observations, high-latency
  segment, packet-loss caveats, and route confidence.
- `active_dependency_path`: dependency group, active member, observation method,
  and confidence.
- `provider_pop_assignment`: provider, endpoint/member, assigned PoP, region,
  and confidence.
- `application_symptom`: app/service, device, symptom, start/end, and severity.
- `user_report`: reported state, reporter/source, affected devices, and symptom
  confidence.
- `operator_observation`: bounded manual observation with explicit assertion
  type and limitations.
- `provider_diagnostic_reference`: URL or copied local reference metadata,
  provider, checked-at time, result state, and limitations.

Freshness behavior:

- Freshness is computed per item relative to telemetry or incident time.
- Stale diagnostics may be retained and displayed but must not raise attribution
  confidence to high.
- Missing diagnostics should be explicit unresolved evidence, not an exception.

Override rule:

- Operator evidence can refine `user_impact`, `operational_risk`, or
  `attribution_confidence`.
- Operator evidence cannot erase sustained telemetry or convert degraded telemetry
  to healthy without a factual diagnostic conflict that is preserved in
  `unresolved_evidence`.

## Backward Compatibility

Current noticeability should remain available temporarily as
`legacy_noticeability` or `legacy_noticeability_reference`. It must be labeled as
legacy once multidimensional artifacts exist.

Old snapshots render unchanged. Existing immutable historical snapshots must not
be opened, regenerated, patched, or rewritten to include new fields.

New renderers should detect multidimensional artifacts by field presence first,
then by recognized model version. Numeric or lexical model-version comparisons
should not drive behavior because model version strings are semantic identifiers,
not ordered versions.

Deprecation path:

- Phase 1: document model and fixtures only.
- Phase 2: add Python evaluator and additive artifacts while keeping current
  noticeability and attribution labels.
- Phase 3: render new dimensions alongside legacy noticeability.
- Phase 4: demote or remove legacy noticeability after comparison and operator
  review.

## Phased Implementation Recommendation

Phase 2 should implement a Python-only evaluator that consumes current telemetry,
provider-neutral dependency metadata, and optional diagnostic evidence. It should

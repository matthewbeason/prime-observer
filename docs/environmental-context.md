# Environmental Context

Environmental Context is Prime Observer's architectural concept for external
systems outside the local home network that may influence the user's network
experience and help reduce uncertainty during observation, attribution, or
investigation.

The repository's current implemented examples of this concept are:

- Internet Conditions, generated by `bin/fetch_cloudflare_radar.py` into
  `viz/internet_conditions.json` and consumed by `viz/index.html`
- APS Power Infrastructure Context, generated by
  `bin/fetch_aps_power_context.py` into `viz/aps_power_context.json` and
  consumed by `viz/index.html`

Environmental Context is evidence only. It adds bounded external context around
local network experience without changing Prime Observer's local-first posture
or moving interpretation into the product.

## What It Is

Environmental Context is:

- optional external evidence gathered by deterministic local Python scripts
- written to local JSON artifacts for renderer consumption
- additive context that may help reduce uncertainty when local evidence alone
  does not fully explain what an operator is seeing
- bounded to factual provider summaries that fit Prime Observer's six dashboard
  questions

Environmental Context may include evidence from domains such as:

- Internet infrastructure
- ISP infrastructure
- Power infrastructure
- Weather
- Regional hazards
- Regional service disruptions

## What It Is Not

Environmental Context is not:

- local telemetry evidence
- an Observation owned by Prime Observer
- Attribution logic
- Health model logic
- a Historical Investigation
- User Noticeability scoring
- prediction, correlation, or recommendation logic
- a generic collection of feeds with unclear operational value

Environmental Context does not tell Prime Observer what happened. It provides
additional external evidence that may help an operator decide how much
uncertainty remains around local measurements and generated conclusions.

## Why It Exists

Prime Observer already answers local observability questions from telemetry,
deterministic health modeling, Observation projection, and historical
investigation evidence.

Some operator questions remain partly uncertain even when those layers are
working correctly:

- Did something broader than the home network appear to be happening?
- Did regional or provider conditions align with the local experience?
- Does external context reduce uncertainty around an upstream-looking issue?

Environmental Context exists so the product can add bounded external evidence
without collapsing Evidence, Observation, Attribution, Investigation, and
external context into one layer.

## Relation To Prime Observer's Core Questions

Environmental Context can support these questions:

- What is happening?
  It may add factual external conditions near the same time window, but it does
  not replace local telemetry or Prime Observer conclusions.
- Why?
  It may narrow uncertainty by showing relevant outside conditions, but it must
  not claim causality stronger than the provider evidence supports.
- Is this unusual?
  It may show whether an external provider reported broad disruption or hazard
  context, but it does not replace Prime Observer's local Pattern Awareness
  model.

This contribution is independent evidence. It can help an operator judge
whether local conditions may have aligned with broader outside conditions, but
it does not perform attribution and does not assert causality.

## Separation From Other Concepts

- Evidence: local telemetry and factual generated summaries remain the source of
  truth for measured network behavior.
- Observation: Prime Observer Observations remain deterministic conclusions it
  owns from local evidence.
- Attribution: Attribution remains Prime Observer logic for classifying local
  versus upstream evidence. Environmental Context may inform operator judgment
  around uncertainty, but it must not participate in attribution decisions.
- Health: the health model remains defined by local telemetry thresholds and
  deterministic classification rules.
- Historical Investigation: investigations remain evidence-first local packages
  for a requested historical window. Environmental Context may be copied in as
  additive evidence when appropriate, but it does not become the investigation
  model itself.
- Noticeability: User Noticeability remains a local deterministic synthesis. It
  must not depend on external provider conditions.

Environmental Context is adjacent to these layers, not a replacement for them.
It may contribute independent evidence alongside Prime Observer's local
artifacts, but the local evidence pipeline remains the architectural center.

## Provider Acceptance Criteria

A provider is a good Environmental Context candidate only if it satisfies all
of these:

- It contributes factual evidence only.
- It can be fetched or generated by deterministic local Python code.
- It writes a local JSON artifact for browser consumption.
- It remains optional and fail-safe.
- It keeps secrets out of browser code.
- It uses additive artifacts rather than replacing existing Prime Observer
  artifacts.
- It stays bounded to information an operator can use to reduce uncertainty.
- It does not create predictions, recommendations, or confidence claims beyond
  the provider evidence.
- It stays separate from Observation, Attribution, Health, and Noticeability
  logic.
- It has a clear time window or generation timestamp.
- It has a meaningful unavailable state.

Good acceptance questions for future agents:

- Does this provider reduce uncertainty around a real Prime Observer question?
- Can the result be expressed as factual context rather than interpretation?
- Can it fail without breaking the dashboard or investigation workflow?
- Would the artifact still make sense if the provider is unavailable?

## Provider Categories

Environmental Context provider categories may include:

- Internet Infrastructure
  Example shape: broad Internet disruption or anomaly summaries.
- ISP Infrastructure
  Example shape: provider-reported regional outage or service-status summaries.
- Power Infrastructure
  Example shape: public outage summaries that may explain home-network
  instability. APS is the current prototype provider in this domain.
- Weather
  Example shape: severe weather summaries that may affect access networks or
  power.
- Regional Hazards
  Example shape: public hazard summaries that may affect access networks or
  power.
- Regional Service Disruptions
  Example shape: public status summaries for major regional services that may be
  confused with network failure.

## Required Architectural Behavior

Environmental Context providers must follow these repository-aligned rules:

- deterministic Python generation
- local JSON artifacts
- renderer-only browser consumption
- optional and fail-safe behavior
- additive architecture
- no secrets in browser code
- no causal claims stronger than evidence supports
- no predictive claims stronger than evidence supports
- graceful unavailable state

Cloudflare Radar / Internet Conditions already demonstrates this pattern:

- generation happens in `bin/fetch_cloudflare_radar.py`
- output is `viz/internet_conditions.json`
- browser consumption happens from generated local artifact data
- missing configuration or provider failure writes an `unavailable` artifact
- explicit configured ASN mode is allowed for provider-scoped traffic anomaly
  checks, but only through deterministic local configuration and additive
  artifact metadata

APS Power Infrastructure now follows the same pattern:

- generation happens in `bin/fetch_aps_power_context.py`
- output is `viz/aps_power_context.json`
- browser consumption happens from generated local artifact data
- provider failure writes an `unavailable` artifact

## Good Candidate Providers

Good candidate providers usually have these traits:

- public or operator-configured summary data with clear timestamps
- broad external scope rather than per-device or per-user detail
- stable evidence that can be summarized without browser secrets
- clear operator value during attribution or investigation uncertainty

Examples:

- Cloudflare Radar Internet Conditions
- APS Power Infrastructure
- public ISP or backbone status summaries, if they can be consumed as bounded
  factual evidence
- public power outage summaries, if they can be represented as regional context
  rather than prediction
- public severe-weather summaries, if they stay tightly scoped to regional
  operational context
- public regional hazard summaries, if they stay tightly scoped to regional
  operational context

## Poor-Fit Providers

Poor-fit providers usually have one or more of these problems:

- they require browser-side authentication or secret exposure
- they return raw logs, per-device details, or user-attribution data
- they imply causal certainty that the evidence cannot support
- they mainly serve alerting, prediction, recommendation, or correlation use
  cases
- they turn Prime Observer into a broad feed reader instead of a bounded local
  observability tool

Examples:

- social-media outage chatter
- generic news feeds
- per-user ISP billing or account dashboards
- raw utility telemetry or control-plane APIs that do not map cleanly to
  summary context
- providers whose main value is forecasting or recommendation rather than
  evidence

## Guidance For Future Agents

- Start from repository boundaries, not from a provider idea.
- Treat Environmental Context as additive evidence, never as new attribution or
  observation logic.
- Reuse the existing optional-provider pattern: deterministic Python generation,
  local JSON artifact, renderer consumption, and unavailable fallback.
- Keep provider scope narrow and operator-first.
- If a provider's role, operational value, or acceptable evidence strength is
  not clearly supported by repository direction, mark it `Needs Matthew Review`.

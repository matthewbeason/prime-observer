# Prime Observer Health Model Audit

Audit date: 2026-06-16

Scope: Wave 4A documented the consistency gap. Wave 4B aligned the dashboard
heatmap and User Noticeability with the canonical composite WAN health model
without changing thresholds, telemetry collection, DNS collection, event
generation, Core Signal, or Olivaw.

## Summary

Prime Observer now has a coherent health model around target classes, WAN
thresholds, sustained degradation, turbulence, LAN elevation, target-group
attribution evidence, the dashboard heatmap, and User Noticeability.

The Wave 4A inconsistency was that canonical WAN bad moments should be
composite WAN evidence across internet and resolver target groups, while the
dashboard heatmap and User Noticeability used internet probes only. Wave 4B
addressed that gap by rendering composite WAN heatmap buckets and feeding User
Noticeability from the same composite WAN target-group model.

## Findings

### Finding 1: Dashboard Heatmap Was Internet-Only

Concept: WAN bad moment.

Wave 4A behavior: the dashboard heatmap used only `internetSeriesMarked`.
Resolver probe sustained bad samples and resolver turbulence were available in
a separate chart and tooltip details, but did not create heatmap buckets.

Expected behavior: under the canonical model, a WAN bad moment is a 15-minute
bucket where one or more WAN target groups, internet or resolver, exhibits
sustained degradation.

Severity: Major.

Wave 4B status: addressed. The dashboard now renders composite WAN buckets and
tooltip evidence distinguishes internet probe, resolver probe, and composite
WAN evidence.

### Finding 2: User Noticeability Used Internet Probes Only

Concept: User Noticeability.

Wave 4A behavior: User Noticeability score and forecast used
`internetSeriesMarked`. Resolver probe degradation, resolver turbulence,
jitter, and loss did not affect the score.

Expected behavior: if User Noticeability is intended to derive from the
canonical health model, it should specify whether it uses composite WAN evidence
or an internet-only subset.

Severity: Major.

Wave 4B status: addressed. User Noticeability now consumes composite WAN p95
samples and uses composite WAN bad moment and turbulence buckets for forecast
evidence. LAN evidence and DNS context remain outside Noticeability inputs.

### Finding 3: Attribution Consumed Evidence Not Represented In The Heatmap

Concept: Attribution Assessment.

Wave 4A behavior: attribution consumed internet probes, resolver probes, and
LAN evidence. The heatmap represented internet probes only.

Expected behavior: an operator should be able to visually reconcile the primary
bad-moment visualization with the evidence used by attribution.

Severity: Major.

Wave 4B status: addressed for WAN target groups. The heatmap now represents
internet and resolver WAN evidence. LAN evidence remains a separate input to
attribution and selected-bucket details, not a WAN target group.

### Finding 4: Persistence Grouping Differs Between Current Export And Investigation Builder

Concept: Sustained Sample.

Current behavior: `bin/transform_latest.py` tracks sustained streaks by path
and `target_class`; `bin/build_investigation.py` tracks them by path,
`target_class`, and host. The dashboard JavaScript also tracks by path and
`targetClass`.

Expected behavior: the canonical model should have one explicit persistence
grouping rule.

Severity: Minor.

Recommendation: in a future implementation wave, choose and document one
grouping rule in code comments or shared test fixtures. The health model in
`docs/health-model.md` defines the canonical level as path plus WAN target
grouping.

### Finding 5: Investigation Events Include Raw/Turbulence Observations As Events

Concept: Investigation Window and WAN Bad Moment.

Current behavior: `bin/build_investigation.py` creates
`wan_bucket_observation` events for buckets with raw bad, sustained bad, or
turbulence evidence.

Expected behavior: investigation events may include factual observations that
are not canonical WAN bad moments, but this distinction must remain explicit.

Severity: Informational.

Recommendation: keep calling these `wan_bucket_observation` events and avoid
renaming all of them to bad moments. The current wording is acceptable.

### Finding 6: WAN Threshold Constants Are Duplicated

Concept: Bad Sample and Turbulence Sample.

Current behavior: thresholds are duplicated in `bin/transform_latest.py`,
`viz/index.html`, and `bin/build_investigation.py`.

Expected behavior: duplicated constants should remain identical or be generated
from one source.

Severity: Minor.

Recommendation: add tests or a shared generated constants contract in a future
implementation wave if threshold drift becomes a risk. The current values
match.

### Finding 7: Pattern Awareness Is Internet-Probe Based

Concept: Pattern Awareness.

Current behavior: Pattern Awareness uses `baseline_p95`,
`baseline_delta_pct`, and `baseline_sample_count` generated from general
internet probe p95 behavior.

Expected behavior: Pattern Awareness should be described as internet-probe
context unless the model is expanded to resolver target groups.

Severity: Informational.

Recommendation: keep Pattern Awareness scoped to "is general internet behavior
unusual for this hour?" unless a later wave explicitly expands the baseline
model.

## Explicit Questions

1. Does the dashboard heatmap represent Internet probes only, or all WAN target
   groups?

   It now represents all WAN target groups: internet probes and resolver
   probes.

2. Should WAN bad moments be Internet-only or composite WAN evidence?

   WAN bad moments are composite WAN evidence: a 15-minute bucket where one or
   more WAN target groups exhibits sustained degradation.

3. Do attribution decisions consume evidence not represented in the heatmap?

   LAN gateway evidence is consumed by attribution and remains separate from
   the heatmap. Resolver-derived WAN evidence is now represented in heatmap
   buckets.

4. Do investigation windows derive from the same health model as attribution?

   Mostly yes. Both use WAN target groups and LAN gateway evidence. Current
   attribution additionally performs classification decisions, while
   investigations emit factual evidence only. Sustained streak grouping differs
   slightly between current export and investigation generation.

5. Does user noticeability derive from the same health model as attribution?

   It now derives from the same composite WAN target-group model for WAN
   evidence. LAN evidence remains outside User Noticeability inputs.

6. Is there any concept currently defined differently in multiple locations?

   Sustained persistence grouping differs between transform/dashboard code and
   investigation generation. The dashboard heatmap, Noticeability, attribution
   exports, and investigations now all use composite WAN target-group evidence
   for WAN bad moments.

7. Is there any visualization that could cause an operator to draw an incorrect
   conclusion?

   The Wave 4A heatmap could look calm when resolver probes were degraded. Wave
   4B addressed that by using composite WAN heatmap buckets and showing
   target-group participation in the tooltip.
